# ADR-010: Money stored as BIGINT whole IDR

- **Status:** Accepted
- **Date:** 2026-07-16

## Context

CrediWise operates in a single currency, Indonesian Rupiah (IDR), which has
no subunit in common practical use (unlike USD cents). Floating-point money
representations are a well-known source of rounding-drift bugs in financial
systems.

## Decision

Store every monetary amount as **`BIGINT` whole rupiah** — no minor-unit
scaling, no floating point (PLAN §4.5 A2, §5.1, §11.1).

## Alternatives Considered

- **Integer minor units (e.g., "sen"):** the conventional pattern for
  currencies with subunits (USD cents), but IDR has no commonly used
  subunit in practice — this would add a scaling factor with no real-world
  referent, purely for pattern-consistency with other currencies CrediWise
  does not support.
- **Floating point (`float`/`double`):** rejected outright — accumulates
  rounding error across balance reconstruction and multi-step engine
  calculations (PLAN §5.1, §15.3).

## Consequences

- Ratios/scores use `Decimal` (28-digit context) internally, rounded
  half-up only at display or final scheduled-payment boundaries (PLAN §5.1,
  §5.7).
- Every engine and repository dealing with money uses `BIGINT` end to end —
  no implicit currency-subunit conversion exists anywhere in the codebase.
- Adding a second currency later requires an explicit new ADR (this decision
  is IDR-specific, not a general "smallest unit" convention).
