# ADR-004: Sync SQLAlchemy 2.0 + Celery

- **Status:** Accepted
- **Date:** 2026-07-16

## Context

The same data-access code runs in two execution contexts: FastAPI request
handlers and Celery pipeline workers (PLAN §8.1, §10.2). A coding-agent-heavy
workflow (CLAUDE.md) benefits from one predictable session/transaction
pattern rather than two.

## Decision

Use **synchronous SQLAlchemy 2.0** (via FastAPI's threadpool for request
handlers) with **Alembic** migrations, and **Celery** for background
processing (PLAN §9, §10.2).

## Alternatives Considered

- **Async SQLAlchemy + ARQ:** lighter-weight and fully async end-to-end, but
  introduces two different session-lifecycle patterns (async context in the
  API, and a separate worker model) that are easier to get subtly wrong
  across many independent PRs/agents. Noted as a future optimization once
  the sync path is stable.

## Consequences

- One `Session`/`Unit of Work` pattern is documented and reused everywhere
  (PLAN §10.2, `app/db/session.py`).
- Request-handler DB calls run in FastAPI's threadpool rather than the event
  loop directly — acceptable for MVP throughput targets (NFR-2).
- Revisiting this decision (a move to async) requires a new ADR and is not a
  silent refactor.
