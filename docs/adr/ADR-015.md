# ADR-015: `SafeBorrowingEngine` ships without `ShockCapacity`; document→assessment pipeline split into `NORMALIZATION` (document-scoped) and `ANALYSIS` (assessment-scoped) stages

- **Status:** Accepted
- **Date:** 2026-07-17

## Context

Sprint 4 (§25, §26.5) implements `NormalizationEngine`, `CashFlowTwinEngine`,
`RiskEngine`, and `SafeBorrowingEngine`. Two structural questions had to be
resolved that PLAN names but does not fully specify:

1. §5.6's formula is `MaximumSafeInstalment = min(BaseCapacity, DSTICapacity,
   WeakestMonthCapacity, TemporalLiquidityCapacity, ShockCapacity)`.
   `ShockCapacity` depends on `ShockEngine`, which does not exist until
   Sprint 5 (§25). Sprint 4 cannot compute it.
2. §8.2's state diagram has `NORMALIZING -> ANALYZING -> COMPLETE` for
   `source_documents`, but `assessments` also need their own lifecycle
   (`assessment_status_enum`, Appendix A) and §11.3 explicitly notes
   `pipeline_stage_runs.assessment_id` is added "once `assessments` exists
   (Sprint 4)" — implying a stage exists that is scoped to an assessment,
   not a document.

## Decision

1. **`ShockCapacity` is omitted from Sprint 4's `min(...)`.** The other four
   terms are fully implemented (`app/engines/safe_borrowing.py`). Because
   `min(...)` can only be lowered by adding a term, never raised, Sprint 4's
   `MaximumSafeInstalment` is a documented conservative upper bound, not an
   overstatement — safe to ship and safe to show to users now, with the
   number only getting *more* conservative once Sprint 5 adds the missing
   term to the same formula. Flagged inline in the engine's module docstring
   and PLAN §26.5/§5.6, not silently claimed complete.

2. **`PipelineStageEnum` gains two members**, not one: `NORMALIZATION`
   (document-scoped — `NormalizationEngine` categorizes one document's own
   transactions, mirrors `EXTRACTION`/`VERIFICATION`) and `ANALYSIS`
   (assessment-scoped — `CashFlowTwinEngine`/`RiskEngine`/`SafeBorrowingEngine`
   run once per assessment across every included document's transactions).
   `pipeline_stage_runs.assessment_id` (migration `0007`) is nullable, mirroring
   the existing nullable `source_document_id`, so one table serves both stage
   classes. Document lifecycle: `DocumentService.confirm()` flips
   `REVIEW_PENDING -> NORMALIZING`, then the dispatched `NORMALIZATION` stage
   flips `NORMALIZING -> ANALYZING`. Assessment lifecycle: `POST /assessments`
   creates the assessment (`PENDING`), freezes the immutable input snapshot
   over the caller's chosen document set, then the dispatched `ANALYSIS`
   stage runs Twin → Risk → SafeBorrowing and flips the assessment to
   `COMPLETE` *and* each included document from `ANALYZING -> COMPLETE`.

## Alternatives Considered

- **Approximate `ShockCapacity` with a placeholder formula now:** rejected —
  any placeholder would be presented to users as if it were the documented
  §5.8 shock-scenario methodology, when no shock scenario has actually run.
  Omitting the term and documenting the gap is more honest than a fake
  number that happens to look plausible.
- **Single `ANALYSIS`-only stage, no `NORMALIZATION`:** rejected — collapsing
  normalization into the analysis stage would mean categorization only runs
  once an assessment is created, so `GET /documents/{id}/transactions`
  (already shipped, Sprint 3) would keep returning `UNKNOWN` categories even
  after the user confirms review, contradicting FR-6's expectation that
  confirmed documents are normalized before being assembled into any
  assessment.
- **Recompute categorization per-document only (not the user's full active
  transaction set):** rejected — `NormalizationEngine`'s internal-transfer
  detection (FR-6 AC5) requires seeing transactions across the user's other
  accounts/documents to find the matching leg of a transfer; scoping to one
  document's rows would silently miss same-day cross-account transfers.

## Consequences

- A dashboard or recommendation shown in Sprint 4 slightly *overstates*
  `MaximumSafeInstalment`/`safe_loan_amount` relative to what Sprint 5 will
  compute once `ShockCapacity` is added — bounded, documented, and expected
  to only tighten (never loosen) in the next cycle.
- `NormalizationEngine` re-categorizes every active transaction the user has
  each time a document reaches `NORMALIZING`, not just the triggering
  document's own rows — more DB writes than a narrower scope, but the only
  way to get instance-correct internal-transfer/recurring detection with the
  current schema; acceptable at MVP scale.
- A document can only ever belong to `ANALYZING` while awaiting inclusion in
  *some* assessment; if a user includes the same document in two different
  assessments (allowed — `assessment_documents` is many-to-many), the first
  completed assessment flips it to `COMPLETE`, and the second assessment's
  analysis stage's "flip included documents to `COMPLETE`" step becomes a
  no-op for that document (guarded on `status == ANALYZING`) — correctly
  idempotent, not an error, but worth remembering if a future reviewer
  wonders why a document's status stops changing after its first assessment.
