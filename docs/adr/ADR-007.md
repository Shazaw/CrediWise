# ADR-007: Deterministic rule engines in MVP (no trained ML model)

- **Status:** Accepted
- **Date:** 2026-07-16

## Context

CrediWise has no real repayment-outcome data yet to train or validate a
default-prediction model, and the hackathon timeframe does not allow for
responsible model development, validation, and fairness testing
(PLAN §1.5 Non-Goals, §19.4).

## Decision

All MVP scoring (Trust Layer, Risk, Shock, Safe Borrowing, Safe Offer) uses
**deterministic, threshold-based rule engines** with versioned weights in
`app/engines/config/model_config.py` (PLAN §5, §15).

## Alternatives Considered

- **Train a lightweight ML model on synthetic data:** would not generalize
  to real repayment behavior and would create a false impression of
  predictive validity without real outcome labels to validate against.

## Consequences

- Every engine interface (`run(inputs, config) -> Result`) is stable enough
  that a trained model can later replace a rule-based engine behind the same
  interface without changing callers (PLAN §4.4, §15.1).
- MVP explainability is maximal — every threshold is a documented, cited
  business rule (PLAN §5), not a learned weight.
- Post-pilot, real repayment outcomes are the trigger to revisit this
  decision (PLAN §1.3 post-MVP metrics).
