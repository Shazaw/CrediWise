# ADR-016: `ShockEngine` design (closed-form `ShockCapacity`, scenario cash-flow model), `OfferEngine` composition, and `RegStatusEnum`/offer-reseeding simplifications

- **Status:** Accepted
- **Date:** 2026-07-17

## Context

Sprint 5 (§25, §26.6) implements `ShockEngine`, `OfferEngine`, seeded
`lenders`, and the full dashboard. Several structural questions had to be
resolved that PLAN names but does not fully specify.

### 1. `ShockCapacity` — the fifth `SafeBorrowingEngine` term (§5.6)

§5.6's formula is `MaximumSafeInstalment = min(BaseCapacity, DSTICapacity,
WeakestMonthCapacity, TemporalLiquidityCapacity, ShockCapacity)`.
`ShockCapacity` is defined in §5.6 as "the highest instalment that remains
at least STRAINED, not DEFICIT, under the configured moderate shock."
ADR-015 (Sprint 4) deferred this term because `ShockEngine` didn't exist
yet.

§8.1's architecture diagram draws the pipeline as `TW -> RK -> SH -> SB ->
OF` (Shock *before* SafeBorrowing), and §15.1 describes `ShockEngine`'s
input as "`FinancialProfile`, proposed instalment, scenario set" — i.e. it
evaluates shocks *against* an instalment that must already exist. This is a
circular dependency: `SafeBorrowingEngine` needs `ShockCapacity` to compute
its final instalment, but `ShockEngine` needs a proposed instalment to
evaluate shocks against.

### 2. `OfferEngine`'s Shock Survivability factor (§5.9)

§5.9 requires "Shock survivability (20%): key shocks rerun with offer
payment events" as one of the eight Safe Offer Score factors. This needs a
resilience score computed against *each offer's own instalment*, not the
recommended instalment.

### 3. `lenders.regulatory_status` member set (§11.3 vs FR-11 AC5)

§11.3 lists `reg_status_enum` as `(REGULATED, UNLISTED)`. FR-11 AC5 requires
"simulated providers use `SIMULATED_REGULATED_PROVIDER`, never an implied
real endorsement" — a label neither of §11.3's two members can express
without misleading copy.

## Decision

### 1. `ShockCapacity` is computed in closed form, inside `SafeBorrowingEngine`, using no `ShockEngine` call

A shock scenario's projected minimum balance is linear in the proposed
instalment: `minimum_projected_balance = savings_buffer +
scenario_free_cash_flow - instalment`. Solving "the highest instalment that
keeps this at or above zero" for the moderate (20% income-drop) scenario has
an exact algebraic solution:

```text
ShockCapacity = max(0, savings_buffer + average_free_cash_flow
                        - median_income * moderate_shock_income_drop_pct)
```

This is computed directly inside `app/engines/safe_borrowing.py`'s `run()`,
using only `SafeBorrowingInput`'s own fields (a new `savings_buffer` field
was added) plus a new `moderate_shock_income_drop_pct` config value (0.20,
shared with `ShockEngine`'s own scenario battery via one constant in
`model_config.py` so the two engines can never disagree about what
"moderate" means). This preserves PLAN §10.1's rule that engines "depend on
nothing but their own inputs and `model_config`, not on each other" — no
search, no iteration, and no dependency on the richer `ShockEngine` at all.

Because `min(...)` can only be lowered by adding a term, this only ever
tightens Sprint 4's numbers (documented there as a conservative upper
bound), never loosens them, exactly as ADR-015 anticipated.

### 2. `ShockEngine` is a separate, fuller battery, run *after* `SafeBorrowingEngine` using its final instalment

`app/engines/shock.py` implements the full 7-scenario battery (§5.8: three
income-drop tiers, delayed dominant income, a fixed emergency expense,
largest-income-source loss, weakest-month replay) plus resilience score and
STRONG/MODERATE/FRAGILE band. `assessment_service.run_assessment_analysis`
runs it once, after `SafeBorrowingEngine`, substituting
`safe_borrowing_result.maximum_safe_instalment` as the proposed instalment —
this is what actually populates `shock_scenarios` rows and
`assessments.shock_resilience_score` for the dashboard's Shock card and
`GET /assessments/{id}/shocks`. `POST /assessments/{id}/simulate` reruns the
same engine with user-supplied overrides (income-drop %, emergency expense,
proposed instalment), returned but never persisted — the canonical battery
in `shock_scenarios` stays the one the `ANALYSIS` stage computed.

Every scenario's cash-flow adjustment is a documented gap-fill (§24.11) —
see `app/engines/shock.py`'s module docstring for the exact formula per
scenario type (income-drop tiers scale `average_free_cash_flow` by the
lost-income amount; `DELAYED_INCOME` charges only `essential_expenses`,
since nothing is permanently lost, just late; `INCOME_SOURCE_LOSS` removes
the single largest-concentration income source's average amount;
`WEAKEST_MONTH_REPLAY` replays the historical minimum as-is).

### 3. `OfferEngine` never calls `ShockEngine` directly; the service layer composes them

`app/services/offer_service.py` runs `shock.run()` once per candidate offer
(substituting that offer's own `instalment_amount`) and passes the resulting
`resilience_score` into `OfferEngine` as a plain `Decimal` scalar
(`shock_resilience_score_for_offer`) — the same composition pattern
`assessment_service.py` already uses for Twin -> Risk -> SafeBorrowing.
`OfferEngine` itself stays a pure function of scalars plus one already-fully-
computed offer (loan-mathematics fields from the new `app/engines/loan_math.py`
pure module — FLAT and REDUCING_BALANCE amortisation schedules, PLAN §5.7).

### 4. `RegStatusEnum` gains a third member, `SIMULATED_REGULATED_PROVIDER`

§11.3's two-member list and FR-11 AC5's explicit labelling requirement both
bind `lenders.regulatory_status`; per §24.11, the acceptance criterion's more
specific requirement wins. The three seeded lenders
(`app/db/seeds/lenders.py`) use `SIMULATED_REGULATED_PROVIDER` (two of
three) and `UNLISTED` (one, to exercise `OfferEngine`'s provider-verification
scoring spread) — none uses bare `REGULATED`, since no MVP provider is a
real regulated entity.

### 5. `POST /assessments/{id}/offers` re-seeding is documented as non-idempotent

Re-calling this endpoint inserts a fresh set of `lender_offers`/
`offer_assessments` rows rather than replacing or deduplicating the prior
set. Offers are simulated, non-financial, re-derivable data (PLAN §6.4:
"simulated lender offers must be visibly labelled simulated"); offer-set
versioning (soft-deleting the prior batch, or returning the existing batch
idempotently) would add meaningful complexity for a demo endpoint the golden
path (§1.6) calls once per assessment. `GET /assessments/{id}/offers`
correctly shows every non-deleted batch ever seeded for that assessment if
the endpoint is called more than once — a known, documented limitation, not
a silently-accepted bug.

## Alternatives Considered

- **Approximate `ShockCapacity` with a bisection search over instalment
  values, actually calling `ShockEngine` per candidate:** rejected — the
  relationship is exactly linear, so a search is strictly more code and CPU
  for an identical answer. It would also violate PLAN §10.1's
  engines-are-mutually-independent rule for no benefit.
- **Give `ShockEngine` an explicit `ShockCapacity`-shaped output so
  `SafeBorrowingEngine` could consume it as a plain scalar the service layer
  passes in (still no direct engine-to-engine call):** considered
  reasonable, but rejected in favor of the closed form because the closed
  form needs none of `ShockEngine`'s other six scenarios computed first —
  it would be strictly more service-layer plumbing (call `ShockEngine` early
  with a not-yet-final instalment, discard six of its seven results) for the
  same one-line formula this ADR already derives directly.
- **Reuse `BandEnum` (HIGH/MEDIUM/LOW) for the top-level Shock Resilience
  Score band instead of a new `ShockResilienceBandEnum`:** rejected — §5.8
  names the band vocabulary explicitly as STRONG/MODERATE/FRAGILE with its
  own 75/50 thresholds (not `BandEnum`'s 80/50), and conflating the two
  would make dashboard copy ambiguous about which threshold scale applies.
  `offer_assessments.shock_resilience_status` *does* reuse `BandEnum`
  because §11.3 explicitly types that column as `band_enum`, not a new type
  — the two enums coexist for different columns, each matching its own
  PLAN-mandated type.
- **Replace/soft-delete prior `lender_offers` on re-seeding:** rejected for
  this cycle as unnecessary complexity — see Decision 5 above.

## Consequences

- `SafeBorrowingEngine`'s `MaximumSafeInstalment`/`safe_loan_amount` are now
  complete per §5.6's documented formula (all five terms implemented) —
  ADR-015's gap is fully closed, not further deferred.
- A user with a very thin `savings_buffer` will see a lower Safe Borrowing
  Capacity than Sprint 4 would have computed for identical transaction data
  — expected and correct per ADR-015's original prediction, not a
  regression.
- `ShockEngine`'s scenario cash-flow formulas are a documented MVP modelling
  choice, not literally derived from a day-by-day liquidity timeline (PLAN
  §15.3 item 5's cross-document consistency gap and this module's own
  monthly-aggregate approximation are the same class of "documented
  simplification, not silently claimed complete" pattern already established
  by prior sprints).
- Calling `POST /assessments/{id}/offers` twice for the same assessment
  leaves two batches of offers visible via `GET .../offers` — acceptable for
  the MVP demo golden path (called once), flagged here for any future
  session that notices duplicate-looking offer rows.
