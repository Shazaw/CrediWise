# ADR-014: Extraction auto-provisions `financial_accounts` when a document has none

- **Status:** Accepted
- **Date:** 2026-07-17

## Context

PLAN §11.3 requires `transactions.financial_account_id` to be `NOT NULL`
("financial_accounts ||--o{ transactions" in the §11.2 ER diagram; the
column is listed without a nullability caveat, unlike the several columns
§11.3 explicitly marks `NULL`). At the same time, `source_documents
.financial_account_id` is nullable, and Sprint 2 (T2.1) deliberately shipped
no `financial_accounts` create/list route — the model's own docstring
anticipated this exact gap: "later sprints (extraction/aggregation)
attribute uploads to an account, whether auto-detected from statement
metadata or explicitly chosen by the user" (`app/models/financial_account.py`).

Sprint 3's extraction stage is the first code that actually needs to create
`transactions` rows, so it is the first point this gap becomes load-bearing:
given a `source_document` with no `financial_account_id`, where does a
newly extracted transaction's `financial_account_id` come from?

## Decision

`app/services/extraction_service.py::_ensure_financial_account` resolves an
account in this order:

1. If the document already has a `financial_account_id` (either
   user-supplied at upload time, or from a prior account/create flow),
   reuse it.
2. Otherwise infer a coarse `account_type` from the document's declared
   `source_type` (`BANK_API`/`SIGNED_STATEMENT`/`ORIGINAL_PDF`/
   `EXPORTED_CSV` → `BANK`; `SCREENSHOT`/`PHOTO` → `EWALLET` — matching the
   golden path's "bank e-statement PDF, or a GoPay screenshot as fallback",
   PLAN §1.6 step 3).
3. Reuse an existing auto-provisioned account of that type for the user if
   one exists (`provider_name = "auto-detected"`), so repeated uploads of
   the same document class don't fragment into duplicate accounts.
4. Otherwise create a new `financial_accounts` row
   (`ownership_status=DECLARED`, `connection_type=UPLOAD`,
   `connection_status=ACTIVE` — all the model's existing defaults) and
   attach it to both the document and the new transactions.

No new HTTP endpoint is added; this is purely a server-side inference that
keeps `transactions.financial_account_id` faithfully `NOT NULL` per §11.3
without forcing a client-facing "create account" step into the MVP upload
flow, consistent with A3's simulated-provider posture (§1.4).

## Alternatives Considered

- **Make `transactions.financial_account_id` nullable:** rejected — this
  would silently diverge from the documented schema (§11.3) rather than
  resolve the documented-but-unbuilt gap the model docstring already named,
  and would push the "which account does this belong to" question further
  downstream (Sprint 4's Twin/aggregation logic) where it is *more*
  disruptive to retrofit, not less.
- **Require the client to create a `financial_accounts` row before
  upload:** rejected for MVP — adds a mandatory extra step to the golden
  path (§1.6) for a decision the system can infer correctly in the common
  case (one statement format per account class), and reopens exactly the
  scope T2.1 deferred.
- **One auto-provisioned account per document instead of per user+type:**
  rejected — would fragment a user's transactions across N accounts for N
  uploads of the same real-world account, breaking Sprint 4's per-account
  aggregation before it even ships.

## Consequences

- `masked_account_number` on an auto-provisioned account is best-effort
  (populated from the extraction engine's `detected_account_holder_name`
  when available, `NULL` otherwise) — it is not a verified account number
  and must not be presented to the user as one.
- A later "create/claim a financial account" flow (if built) should offer a
  path to re-attribute an auto-provisioned account's transactions to a
  user-confirmed one; that migration is out of scope for Sprint 3.
- If a user's statements legitimately span multiple real bank accounts of
  the same declared `source_type` (e.g. two different `ORIGINAL_PDF` bank
  accounts), they currently collapse into one auto-provisioned account.
  Distinguishing them requires either account-number extraction (not
  implemented) or a user-facing account-selection step — tracked as a
  post-Sprint-3 follow-up, not a Sprint 3 exit-criterion gap.
