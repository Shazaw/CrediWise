# ADR-005: PostgreSQL 16 + JSONB

- **Status:** Accepted
- **Date:** 2026-07-16

## Context

CrediWise's data model is fundamentally relational and audit-critical
(immutable lineage across documents, processing runs, transactions,
snapshots, and scores — PLAN §11), but several payloads (reason-code
evidence, consent data-scopes, flags) are naturally flexible/semi-structured.

## Decision

Use **PostgreSQL 16** as the system of record, with `JSONB` columns for
flexible payloads (`flags_json`, `data_scope_json`, `evidence_json`, etc.)
(PLAN §9, §11.1).

## Alternatives Considered

- **MongoDB:** better fit for pure document flexibility, but weaker
  relational-integrity guarantees for a system whose core value proposition
  is an auditable financial lineage (users → accounts → documents →
  transactions → assessments → offers, all FK-constrained with
  `ON DELETE RESTRICT` on financial lineage, PLAN §11.1).

## Consequences

- Strong FK/constraint enforcement protects audit integrity by default.
- `JSONB` gives schema flexibility exactly where it is needed (reason codes,
  scopes) without abandoning relational guarantees elsewhere.
- Postgres-native `ENUM` types are used for closed sets (status, category,
  band, role) rather than free-text columns.
