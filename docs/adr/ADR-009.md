# ADR-009: S3-compatible object storage + presigned URLs

- **Status:** Accepted
- **Date:** 2026-07-16

## Context

Raw uploaded financial documents, page renders, and (post-MVP) exports must
be stored securely, cheaply, and without locking the project to one cloud
vendor (NFR-15).

## Decision

Use **S3-compatible object storage** (Cloudflare R2 / AWS S3 in
staging/production, MinIO locally via docker-compose) behind a `StoragePort`
interface, accessed only through **short-lived presigned URLs** — the app
never holds long-lived storage credentials (PLAN §9, §16.1, §17.3).

## Alternatives Considered

- **Store documents directly in Postgres (bytea):** simplest to wire, but
  bloats the database, complicates backups, and has no native
  presigned-access or lifecycle/retention tooling.
- **Vendor-specific SDK (e.g., AWS-only):** would work, but creates lock-in
  that contradicts the portability requirement (NFR-15).

## Consequences

- Local dev, CI, staging, and production all use the same `StoragePort`
  interface; only the endpoint/credentials differ (PLAN §17.3, §20.1).
- Bucket layout separates raw/processed/exports (`raw/{user_id}/{document_id}`,
  etc.) so lineage stays legible (PLAN §17.3).
- Retention/erasure jobs operate against the same interface regardless of
  which provider is configured (PLAN §11.5).
