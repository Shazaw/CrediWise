# ADR-013: File-security stage runs synchronously in `POST /documents`

- **Status:** Accepted
- **Date:** 2026-07-17

## Context

PLAN §8.2's processing-pipeline state diagram shows `UPLOADED -> SECURITY_CHECK`
as a stage the async `documents` Celery worker performs, alongside OCR/verify/
normalize (§8.3's sequence diagram: `W->>W: security -> OCR -> verify ->
normalize`). FR-3 (Document Upload & File Security), however, imposes two
constraints that a worker-side security stage cannot satisfy:

1. **AC3 (dedup):** a re-upload of the same SHA-256 hash by the same user
   must return the *existing* `source_document` — with no second row and no
   second object-storage write (CLAUDE.md's named measurable outcome). That
   decision has to be made before any row is created, i.e. before anything
   is handed to a worker.
2. **AC4 (password-protected PDFs) + §24.10:** a PDF password may only ever
   exist in request-scoped process memory. It must never be persisted,
   logged, or — critically — passed as a Celery task argument, since Celery
   task payloads are serialized through Redis.

If the security stage (MIME/magic-bytes/size/hash/dedup/password/script-scan)
ran inside the `documents` worker as the diagram's literal sequencing
suggests, either the password would have to travel through Redis (violating
§24.10), or dedup would have to be discovered *after* a row and a storage
write already existed (violating AC3 and CLAUDE.md's stated outcome).

## Decision

The entire FR-3 file-security stage (`app/engines/file_security.py`) runs
synchronously inside the `POST /documents` request handler
(`app/services/document_service.py`), before any `source_documents` row is
created and before any bytes reach object storage:

1. Hash the upload; look up an existing document for `(user_id, file_hash)`.
   A match returns immediately — no row, no storage write.
2. Otherwise run the full security engine (size/magic-bytes/PDF password
   decrypt+action-scan/image bomb checks) against the in-memory bytes.
   PDF passwords are used here and only here, then discarded.
3. Create the `source_documents` row with the resulting terminal-or-passing
   status; store bytes only when the outcome is `PASSED`.
4. Only then dispatch `process_document` onto the `documents` Celery queue,
   passing nothing but `document_id`.

`app/pipeline/document_tasks.py`'s `process_document` task is scoped down to
exactly the state diagram's `SECURITY_CHECK -> EXTRACTING` edge: the document
handed to it has already passed validation, so the task's job is recording
that it has now entered the async pipeline. This is the literal Sprint 2 exit
criterion ("uploading a fixture PDF stores it, dedups on re-upload, advances
to EXTRACTING") and the seam Sprint 3's real OCR/extraction worker extends.

## Alternatives Considered

- **Run the security stage in the worker, pass the password as a task arg:**
  rejected outright — directly violates §24.10 and NFR-6/§18.5's password
  handling rules.
- **Run the security stage in the worker, hold the password in a short-TTL
  Redis key instead of the task payload:** still stores the password in
  Redis (even briefly), which is exactly what §17.2 ("Never cache: raw
  documents, tokens, or PII") and §24.10 rule out; adds a second stateful
  side-channel for no benefit over just doing the work synchronously.
- **Do dedup synchronously, defer the rest of the security stage (mime/size/
  script-scan) to the worker:** avoids the password problem partially (still
  need to synchronously handle at least the password check) while splitting
  one conceptual validation step across two execution contexts for no
  reason; more moving parts, no clearer correctness story.

## Consequences

- `POST /documents` does strictly more synchronous work than the sequence
  diagram implies (hashing, MIME/magic-byte checks, PDF parsing, image
  decoding) — all fast, CPU-bound, and bounded (`MAX_UPLOAD_MB` caps the
  read), so this stays well within NFR-1's pipeline latency budget; it is
  not subject to NFR-2 (that budget explicitly excludes pipeline-triggering
  endpoints).
- The `documents` queue/worker exists from Sprint 2 as scaffolding for
  Sprint 3's real OCR/extraction stage, not because Sprint 2 itself needs
  meaningful async work — this is intentional, matching how `model_versions`/
  `model_config` were scaffolded in Sprint 1 ahead of real engine weights.
- Any future stage of the pipeline that needs a user-supplied secret must
  follow the same rule: handle it synchronously in the request that received
  it, never in a queued task.
