# ADR-012: Model config as versioned code + config_hash

- **Status:** Accepted
- **Date:** 2026-07-16

## Context

Scoring weights and thresholds (PLAN §5) will change over time as the
product is tuned. Historical assessments must never silently change meaning
when that happens (NFR-17, FR-18) — a regulator or a disputing user must be
able to reproduce exactly how a past score was computed.

## Decision

Store engine weights/thresholds as **versioned Python code**
(`app/engines/config/model_config.py`), keyed by `model_version`, with a
`config_hash` (hash of that file) stamped on the `model_versions` table and
on every assessment that used it (PLAN §17.4, §19.2).

## Alternatives Considered

- **Store weights as mutable DB rows:** easier to tweak at runtime (no
  deploy needed), but makes "what config produced this historical score"
  unanswerable without a separate audit table duplicating the same
  versioning problem, and risks accidental in-place edits to historical
  config.

## Consequences

- Exactly one `ACTIVE` version exists per model name at a time (partial
  unique index, PLAN §11.3 `model_versions`).
- Any weight/threshold change is a new `model_version` plus a required
  golden-fixture review (PLAN §15.4, §21.3, CLAUDE.md §7.4) — never an
  in-place edit to historical config (CLAUDE.md §7.4).
- Re-running an old `assessment_input_snapshot` under its original
  `model_version` reproduces bit-identical output (PLAN §15.4, FR-18 AC1).
